As 6 Vantagens da Nuvem

  • 1. Trocar CapEx por despesa variável — em vez de comprar servidores, você paga só pelo que usa
  • 2. Economias massivas de escala — a AWS compra hardware em volume global; você herda esse desconto
  • 3. Parar de adivinhar capacidade — escale conforme a demanda real, sem superprovisionamento
  • 4. Aumentar velocidade e agilidade — recursos em minutos vs semanas com hardware físico
  • 5. Parar de gastar com data centers — sem custo de energia, refrigeração, segurança física, manutenção
  • 6. Ir global em minutos — lançar em outras regiões do mundo com poucos cliques

4 Conceitos de Infraestrutura (muito cobrados)

ConceitoDefiniçãoPalavra-chave
EscalabilidadeCapacidade de crescer (ou diminuir) conforme necessáriopode crescer
ElasticidadeEscala automática para cima E para baixo conforme a demandaautomático, pico e vale
Alta DisponibilidadeSistema continua operando com mínimo de downtimeminimiza interrupção
Tolerância a FalhasSistema continua operando SEM interrupção mesmo com falhaszero downtime
Armadilha recorrente: nenhum serviço AWS garante 100% de disponibilidade ou "nunca falha". Alternativas com essa linguagem são sempre erradas.

Escala Vertical vs Horizontal

  • Vertical (scale up) — máquina MAIOR: mais CPU, mais RAM na mesma instância
  • Horizontal (scale out) — MAIS máquinas: adicionar instâncias em paralelo
Modelo de Responsabilidade Compartilhada
AWS cuida DA nuvem — você cuida NA nuvem
ver conteúdo →

AWS é responsável por ("segurança DA nuvem")

  • Hardware físico, datacenters, energia, refrigeração
  • Infraestrutura global (regiões, AZs, edge locations)
  • Software de virtualização (hypervisor)
  • Rede física subjacente

Cliente é responsável por ("segurança NA nuvem")

  • Dados do cliente (criptografia, backup)
  • Gerenciamento de identidades e acessos (IAM)
  • Configuração de rede (Security Groups, NACLs)
  • Sistema operacional de instâncias EC2 (patches, atualizações)
  • Aplicações e código

Modelos de Serviço
IaaS, PaaS, FaaS, SaaS — exemplos práticos na AWS
ver conteúdo →
ModeloVocê gerenciaAWS gerenciaExemplo AWS
IaaSSO, runtime, app, dadosHardware, rede físicaEC2
PaaSApp, dadosSO, runtime, middlewareRDS, Elastic Beanstalk
FaaSApenas o código da funçãoTudo (servidor, SO, runtime)Lambda
SaaSNada — só usa o softwareTudoGmail, Salesforce

Como identificar na prova

  • EC2 = IaaS — você instala o SO, faz os patches, instala o runtime. A AWS te deu só o ferro.
  • RDS = PaaS — você não acessa o SO. A AWS cuida do banco, patches, backups. Você só usa o banco.
  • Lambda = FaaS — você escreve a função. Não existe servidor fixo. A AWS executa quando necessário e cobra por milissegundo.
Regra mental: quanto mais baixo na lista, menos você gerencia. SaaS = você gerencia nada.

Users, Groups, Roles, Policies — root user, MFA, least privilege
ver conteúdo →

Componentes do IAM

ComponenteO que éQuando usar
UserRepresenta uma pessoa ou aplicação com credenciais permanentesDesenvolvedor, conta de serviço
GroupColeção de usuários — você aplica políticas no grupoEquipe de devs, equipe de admins
RoleIdentidade temporária assumida por serviços ou usuários externos. Sem senha fixa.EC2 acessando S3, usuário de outra conta
PolicyDocumento JSON que define o que é permitido ou negado. Anexa a usuário/grupo/role.Sempre — define as permissões

Root User

  • Conta criada ao abrir a AWS — tem acesso irrestrito e irrevogável a tudo
  • Nunca usar no dia a dia — criar um usuário IAM para operações normais
  • Deve ter MFA habilitado imediatamente
  • Uso aceitável: fechar a conta, alterar plano de suporte, recuperar acesso bloqueado

Princípio do Menor Privilégio (Least Privilege)

Conceder apenas as permissões mínimas necessárias para a tarefa. Se um usuário precisa ler S3, ele não deve ter permissão de deletar EC2.

IAM é gratuito e global — não é por região. As políticas criadas valem para toda a conta.
Roles vs Users: a prova frequentemente pergunta "qual é a forma correta de dar acesso ao EC2 para acessar o S3?". Sempre Roles, nunca embutir credenciais de usuário no servidor.

AWS Organizations

Permite gerenciar múltiplas contas AWS sob uma estrutura hierárquica — uma conta root (management) e contas-membro organizadas em Unidades Organizacionais (OUs).

Service Control Policies (SCPs)

  • Políticas que definem o que as contas-membro podem e não podem fazer
  • Aplicadas na OU ou na conta — afetam todos os usuários, inclusive administradores da conta-membro
  • Não dão permissões — apenas limitam o que é possível fazer

Cobrança Consolidada

  • Uma única fatura para todas as contas da organização
  • Descontos por volume são calculados somando o uso de todas as contas
  • Benefício imediato: economias de escala internas
Na prova: "empresa quer garantir que nenhuma conta-membro possa criar recursos em determinada região" → SCP

KMS vs CloudHSM

  • KMS — gerenciado pela AWS, mais simples, mais barato. A AWS tem acesso às chaves. Para maioria dos casos.
  • CloudHSM — hardware físico dedicado, você controla as chaves, a AWS não tem acesso. Para regulações rigorosas (PCI, FIPS).

CloudTrail está habilitado por padrão para os últimos 90 dias de atividade de gerenciamento. Para retenção maior, você cria um trail que grava em S3.

At-Rest vs In-Transit

TipoO que éComo proteger
At-restDados armazenados (no S3, EBS, RDS)KMS, criptografia de volume/objeto
In-transitDados em movimento (entre cliente e servidor, entre serviços)SSL/TLS, HTTPS, certificados ACM

AWS Certificate Manager (ACM)

Emite e gerencia certificados SSL/TLS de forma gratuita. Integra com ALB, CloudFront e API Gateway. Cuida da renovação automática — você não precisa gerenciar certificados manualmente.

Na prova: "empresa quer garantir que dados em trânsito sejam criptografados" → SSL/TLS / ACM. "Dados armazenados no S3 devem ser criptografados" → KMS / Server-Side Encryption.

Os 3 tipos de storage que a prova confunde

ServiçoTipoAnalogiaQuando usar
S3Object StorageDropbox giganteArquivos, backups, imagens, logs, sites estáticos
EBSBlock StorageHD externo da instânciaDisco de uma instância EC2 — persiste após desligar

Classes do S3 (do mais caro ao mais barato)

  • S3 Standard — acesso frequente, alta durabilidade, mais caro
  • S3 Intelligent-Tiering — move automaticamente entre tiers conforme acesso
  • S3 Standard-IA — acesso infrequente, mais barato para armazenamento
  • S3 Glacier / Glacier Deep Archive — arquivamento longo prazo, horas para recuperar, muito barato
S3 não é sistema de arquivos — é object storage. Não serve para montar como disco de um servidor.

Cobrança Consolidada
Organizations para consolidar faturas e obter descontos por volume
ver conteúdo →

Com AWS Organizations, todas as contas-membro recebem uma única fatura consolidada na conta de gerenciamento.

Benefícios

  • Visibilidade — um lugar para ver todos os custos
  • Descontos por volume — o uso de todas as contas é somado para atingir faixas de desconto
  • Compartilhamento de reservas — Reserved Instances e Savings Plans de uma conta beneficiam todas as outras
Na prova: "empresa quer reduzir custo combinando o uso de S3 de 10 contas diferentes" → cobrança consolidada via Organizations.

Erros mais comuns na prova

Pegadinhas clássicas que derrubam quem estudou mas não prestou atenção nos detalhes.

⚠️
"Nunca falha" e "100% de disponibilidade" são sempre errados
Nenhum serviço AWS garante disponibilidade absoluta. Alta disponibilidade e tolerância a falhas reduzem o impacto de falhas, não as eliminam. Se a alternativa diz "garante que nunca haverá interrupção" — é errada.
⚠️
Patch do SO: EC2 = cliente / RDS = AWS
No EC2 (IaaS), você controla o sistema operacional — você aplica os patches. No RDS (PaaS), a AWS gerencia o SO e o banco — você nunca acessa o SO do RDS. Essa distinção cai em pelo menos 2 questões.
⚠️
IAM Roles ≠ IAM Users
User tem credenciais permanentes. Role é assumida temporariamente. Quando a prova pergunta "como um EC2 deve acessar o S3 de forma segura?" a resposta é Role — nunca credenciais de usuário embutidas no servidor.
⚠️
S3 ≠ EBS ≠ EFS (três tipos completamente diferentes)
S3 = object storage (arquivos, backups, imagens). EBS = disco de bloco para uma instância EC2. EFS = sistema de arquivos compartilhado entre múltiplos EC2. Confundir esses três é o erro mais clássico de storage.
⚠️
High Availability ≠ Fault Tolerance
Alta Disponibilidade = minimiza downtime (pode haver breve interrupção). Tolerância a Falhas = zero downtime, sistema continua funcionando mesmo com falha ativa. Multi-AZ = HA. Active-Active multi-região = FT.
⚠️
CloudWatch vs CloudTrail (confusão clássica)
CloudWatch = monitoramento de métricas e logs (CPU alta, alarme de erro, performance). CloudTrail = auditoria de ações de API (quem deletou o bucket, quem alterou a política). Pergunta "quem fez isso?" = CloudTrail.
⚠️
Spot Instances podem ser interrompidas
Spot dá até 90% de desconto, mas a AWS pode retomar a instância com 2 minutos de aviso. Nunca use Spot para workloads críticos que não toleram interrupção (banco de dados, processos sem checkpoint).
⚠️
Trusted Advisor completo só em Business ou superior
O plano Basic e Developer têm acesso limitado ao Trusted Advisor (apenas 7 verificações de segurança). As centenas de recomendações completas (custo, performance, limites) requerem Business, Enterprise On-Ramp ou Enterprise.
